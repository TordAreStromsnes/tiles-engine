# Issue Shaper

Use this skill before implementation work that affects product behavior,
architecture, formats, editor UX, runtime systems, or GitHub process.

## Issue Structure

Every issue should include:

- Title.
- Labels.
- Project fields.
- Problem.
- Why now.
- Scope.
- Out of scope.
- Proposed approach.
- Acceptance criteria.
- Definition of done.
- Risks and open questions.
- Test expectations.

## Rules

- Keep issues small enough to review.
- Split research from implementation when the answer is unknown.
- Include `tiles-engine` on every issue.
- Include one `type:*`, one `area:*`, and one `priority:*` label.
- Use `risk:high` when a decision affects multiple future systems.
- Do not hide future work inside acceptance criteria. Create follow-up issues.

## Ready Checklist

An issue is ready when:

- The problem is specific.
- The reason to do it now is clear.
- Success can be verified.
- Out-of-scope boundaries are explicit.
- Definition of done can be audited.
